0. Sesión start — antes de cualquier trabajo no trivial

Antes de un /opsx:apply, un edit de código, una decisión arquitectónica o un commit, el agente invoca scripts/preflight.ps1 y reporta el output al operador en chat. Sesiones triviales (preguntas, exploración) pueden saltarlo — juicio del agente. No hay SessionStart hook: se instaló en v1 y se revirtió tras dogfooding (overhead per-sesión, output invisible al operador, escapes cross-shell frágiles). 

powershell.exe -NoProfile -ExecutionPolicy Bypass -File scripts/preflight.ps1

Las 6 chequeos de preflight

#CheckTag típicoNotas
1Active changes count[OK] 0–1, [WARN] >1 Cuenta dirs en openspec/changes/ excluyendo archive/
2backend/mvnw -q compile[OK] / [FAIL] Fast compile, NO verify. Si falla, el agente debe parar y preguntar.
3Working tree cleanliness[OK] / [WARN] git status --porcelain. Sucio es WARN, no FAIL.
4Branch + last commit[OK] informativo Solo contexto, nunca falla.
5Artefactos por change[OK] / [FAIL] Verifica presencia de proposal.md, design.md, tasks.md, progress.md en cada active change.
6Supabase backup freshness[OK] / [WARN] / [SKIP] Lee r2:my-finance-view-backups/status/last-success.json con rclone; [SKIP] si no instalado.
Semántica de exit code: exit = número de [FAIL]. [WARN] y [SKIP] no fallan. Un [FAIL] mvn compile failed obliga al agente a parar y preguntar al operador antes de empezar /opsx:apply.

El otro artefacto nuevo — progress.md

Cada openspec/changes/<id>/ carga un progress.md en YAML (6 keys) que el subagente backend-developer reescribe al cerrar cada tarea. Es el anchor que sobrevive a compaction: la próxima sesión lo lee y reconstruye contexto en una sola Read. 

# progress.md — schema v1 (YAML)
current_task: "3.2"
last_completed: "3.1"
next_step: "implementar AuthFilter unit test antes de wiring"
decisions_pending_design_update: []
blockers: []
last_updated: "2026-05-31T14:23:00Z"
  • Cadencia: per-task, NO per-tool-call (Decision 2 del design).
  • Authority split: tasks.md checkboxes = "qué está hecho"; progress.md = "qué hacer next + decisiones mid-flight".
  • Staleness: si last_updated < HEAD, adversarial-reviewer emite Minor finding process-tooling.
  • Recovery: si falta, /opsx:apply lo crea desde openspec/templates/progress-template.md y loguea [recovered missing progress.md].

1. Líder & subagentes — quién hace qué

El líder es el hilo principal de Claude en el contexto principal. Orquesta — no implementa. Toda escritura de código Java pasa por un subagente. Esta separación es la que permite resúmenes de contexto manejables y trabajo paralelo seguro.

El líder (hilo principal)

Modelo: Opus 4.7 — vive todo el ciclo
  • Corre preflight.ps1 al inicio de sesiones no triviales.
  • Maneja los 4 gates (A/B/D/C) — una decisión a la vez vía AskUserQuestion.
  • Lee progress.md en Gate D paso 0 y postea "resuming from".
  • Triage findings adversariales contra el threat model.
  • Spawn subagentes con el modelo, batch y modo (async / worktree) correctos.
  • Cierra con /commit → /opsx:archive → /openspec-sync-specs → /update-docs.

backend-developer

Modelo: Sonnet 4.6 (pinned) — vive durante Fase 5
  • Implementa tasks de tasks.md con TDD por tarea.
  • Reescribe progress.md al cerrar cada tarea — refresca last_updated, mueve current_task, marca blockers si surge alguno.
  • Conoce el modular-monolith-by-domain, jOOQ over JPA, BigDecimal HALF_EVEN, ProblemDetail, Testcontainers.
  • Rechaza CleanArch-by-layers, hexagonal maximalist, CQRS, JPA, WebFlux.
  • Mirror de la directiva preflight en su definición — corre preflight.ps1 al inicio de su propia sesión.

adversarial-reviewer

Modelo: heredado — vive durante Fase 6
  • Red-team aislado del change antes de merge.
  • Output: archivo markdown (adversarial-review.md), terse stdout.
  • Severidad Blocker / Major / Minor / Question.
  • Briefed con el threat model verbatim de proposal.md.
  • Checks específicos: progress.md staleness, missing-preflight evidence → Minor process-tooling.

Otros subagentes (on-demand)

Modelo: variable — vive durante una fase específica
  • code-auditor — milestone-level audit, end-of-épica.
  • product-strategy-analyst — ideación / value-prop antes de /enrich-us.
  • Explore / general-purpose — research fan-out en Fase 2.
  • frontend-developer — placeholder, INACTIVE hasta que arranque el frontend.
Regla de oro: si pides "actúa como backend-developer", el líder DEBE spawnear el subagente — nunca rolear en main context. Esto está fijado en CLAUDE.md y en la memoria feedback-opsx-apply-via-backend-developer.

Cada fase azul es trabajo del agente. Cada barra roja es un gate HITL obligatorio: Claude se detiene hasta que el operador da OK explícito.

Por qué A → B → D → C: los letters son estables (para cross-reference); el orden por fase es A (3), B (4), D (5), C (6). C dispara después de D porque la review es post-implementación.
Excepción — commit-directo sin OpenSpec: refactors mecánicos sin design decisions (ej. git mv masivo, rename de paquetes, path updates en docs/CI) saltan el flujo OpenSpec. Patrón: design doc en plans/<YYYY-MM-DD>-<slug>.md como commit 0 → commits temáticos en branch → PR. Los gates A/B/D/C no aplican porque no hay spec que debatir; el design doc + PR description proveen el audit trail. Precedente: monorepo-restructure (2026-05-31). Si dudas si tu trabajo califica, asume que SÍ requiere OpenSpec y deja que el operador te redirija.

3. Los 4 Gates en detalle

Gate A — Decisiones de arquitectura

Fires: después de Fase 1, antes de escribir design.md
Claude hace
  • Lista cada decisión arquitectura-shaping (storage, scheduler, encryption, alerting, host, modelo de persistencia, security boundary…)
  • Para cada decisión, presenta 2-4 opciones con trade-offs vía AskUserQuestion — una a la vez, nunca bundling de 12
  • Registra cada decisión cerrada en design.md §Decisions con alternativas rechazadas + por qué
Operador hace
  • Responde. Si quiere diferir, dice [DEFERRED] explícito.
Pasa cuando: cero decisiones abiertas. Nada que Claude quiera asumir queda sin aprobar.

Gate B — Plan de tareas

Fires: después de design.md, antes de tasks.md
Claude hace
  • Bulleted plan: N top-level tasks, una frase cada uno, en orden de implementación
  • Marca cuáles son independientes (paralelizables) vs secuenciales
  • Marca cuáles son operator-manual vs agent-implementable
  • Estimación relativa S/M/L (no horas)
Operador hace
  • Trim / reorder / split / merge. Cortar scope aquí cuesta 10× menos que en Gate C.
Pasa cuando: operador dice "go" o "go con estos edits".

Gate D — Comenzar implementación

Fires: entre tasks.md completo y el primer spawn de backend-developer
Claude hace
  • Paso 0 (nuevo desde 2026-05-27): lee openspec/changes/<id>/progress.md y postea un párrafo "resuming from" citando current_task, next_step, y cualquier decisions_pending_design_update o blockers no vacíos.
  • Si last_updated < HEAD commit timestamp, marca staleness explícitamente: WARNING: progress.md last updated <T1> but most recent commit is <T2>.
  • Si progress.md no existe, lo crea desde openspec/templates/progress-template.md y loguea [recovered missing progress.md].
  • Confirma: "Voy a spawn backend-developer en Sonnet 4.6 para tasks X.Y → X.Z. Async o secuencial?"
  • Nombra cada subagente que va a spawn y el rango de tasks.
  • Surfaces los manual prereqs del operador (ej: "task 1.2 — tienes que correr age-keygen antes").
Operador hace
  • Confirma, redirecciona (otro subagente, otro modelo), o pide splittear el apply en batches.
  • Si el "resuming from" no cuadra con su modelo mental, corrige antes de dar "go".
Pasa cuando: operador dice "go".

Gate C — Triage findings adversarial

Fires: después del reporte del adversarial-reviewer, antes de incorporar findings
Claude hace
  • Lee el threat model de proposal.md
  • Auto-clasifica cada finding contra severidad + threat model
  • Presenta solo las excepciones (Minor/Question que sugiere aceptar, Major fuera de threat model que sugiere rechazar)
Operador hace
  • Aprueba las excepciones. Confía los defaults.
Pasa cuando: cada finding tiene disposition (incorporated / deferred / rejected (out of threat model)) y el operador revisó la lista de excepciones.

4. Severidad de findings — política de triage

SeveridadDefault actionOperador OK necesario?
Blocker Incorporar en este change No (solo notificación)
Major Incorporar salvo que el threat model lo excluya Sí, solo si rechaza
Minor Diferir a follow-up change Sí, solo si acepta ahora
Question Responder en comentario, sin code change Sí, solo si acepta como code change
Threat model gate: cada proposal.md DEBE declarar un ## Threat model nombrando el adversario y qué está fuera de scope. Findings fuera de threat model se auto-rechazan, con razón de una línea en design.md §Rejected findings.

Ejemplo aplicado — supabase-backup-policy

## Threat model (declarada en proposal.md)
Adversary: opportunistic cloud-storage breach.
Out of scope: local forensic adversary with physical disk access
            — therefore `cipher /W`, VSS clear, Tails live-USB findings auto-rejected.

5. Async / paralelo — cuándo aprovechar subagentes en paralelo

Paralelo ≠ más barato. 3 subagentes 5 min = 3× tokens de 1 subagente 5 min. Paralelo compra wall-clock cuando estás bloqueado, no presupuesto. Default: secuencial. Opt-in a paralelo en Gate D.

Fase 2 — Explorar
Research fan-out
Múltiples Explore subagentes en distintas sub-preguntas.
Single message → N Agent calls in parallel
Fase 5 — Implementar
Tasks independientes
Si tasks.md tiene grupos sin overlap (ej: endpoint A vs endpoint B), spawn N backend-developer con isolation: "worktree". Reconciliar mergeando branches.
Fase 6 — Adversarial review
Background mientras drafteas
Spawn adversarial-reviewer con run_in_background: true. Mientras corre: redactar commit message, actualizar Notion, draftear /update-docs deltas.
Fase 8 — Cierre
Sync + docs en paralelo
/openspec-sync-specs y /update-docs tocan archivos distintos. Dos Agent calls en paralelo (cuidado con conflictos si el change tocó SPEC.md y specs/).
Cuándo NO paralelizar: si quieres ver el trabajo pasar para poder interrumpir, secuencial es mejor — más fácil interjeccionar.

7. Cuándo interrumpir

Si Claude no se detiene en un gate, interrúmpelo y dilo.

Los gates son el contrato. Saltarse uno sin OK del operador es un bug del proceso, no una feature. Frases útiles:
  • "Corre preflight.ps1 antes de tocar nada." (Sesión start)
  • "Antes de spawn, postea el resuming-from del progress.md." (Gate D paso 0)
  • "Espera — antes de design.md, lista las decisiones abiertas." (Gate A)
  • "Antes de tasks.md, muéstrame el plan en bullets." (Gate B)
  • "No spawnees backend-developer todavía — confirma el batch primero." (Gate D)
  • "Findings: dame solo las excepciones a la política default." (Gate C)
  • "No implementes en main context — spawn el subagente." (regla líder vs subagentes)
