2. Los 4 Gates en detalle

Gate A — Decisiones de arquitectura

Fires: después de Fase 1, antes de escribir design.md
Claude hace
  • Lista cada decisión arquitectura-shaping (storage, scheduler, encryption, alerting, host, modelo de persistencia, security boundary…)
  • Para cada decisión, presenta 2-4 opciones con trade-offs vía AskUserQuestion — una a la vez, nunca bundling de 12
  • Registra cada decisión cerrada en design.md §Decisions con alternativas rechazadas + por qué
Operador hace
  • Responde. Si quiere diferir, dice [DEFERRED] explícito.
Pasa cuando: cero decisiones abiertas. Nada que Claude quiera asumir queda sin aprobar.

Gate B — Plan de tareas

Fires: después de design.md, antes de tasks.md
Claude hace
  • Bulleted plan: N top-level tasks, una frase cada uno, en orden de implementación
  • Marca cuáles son independientes (paralelizables) vs secuenciales
  • Marca cuáles son operator-manual vs agent-implementable
  • Estimación relativa S/M/L (no horas)
Operador hace
  • Trim / reorder / split / merge. Cortar scope aquí cuesta 10× menos que en Gate C.
Pasa cuando: operador dice "go" o "go con estos edits".

Gate D — Comenzar implementación

Fires: entre tasks.md completo y el primer spawn de backend-developer
Claude hace
  • Confirma: "Voy a spawn backend-developer en Sonnet 4.6 para tasks X.Y → X.Z. Async o secuencial?"
  • Nombra cada subagente que va a spawn y el rango de tasks
  • Surfaces los manual prereqs del operador (ej: "task 1.2 — tienes que correr age-keygen antes")
Operador hace
  • Confirma, redirecciona (otro subagente, otro modelo), o pide splittear el apply en batches
Pasa cuando: operador dice "go".

Gate C — Triage findings adversarial

Fires: después del reporte del adversarial-reviewer, antes de incorporar findings
Claude hace
  • Lee el threat model de proposal.md
  • Auto-clasifica cada finding contra severidad + threat model
  • Presenta solo las excepciones (Minor/Question que sugiere aceptar, Major fuera de threat model que sugiere rechazar)
Operador hace
  • Aprueba las excepciones. Confía los defaults.
Pasa cuando: cada finding tiene disposition (incorporated / deferred / rejected (out of threat model)) y el operador revisó la lista de excepciones.

3. Severidad de findings — política de triage

SeveridadDefault actionOperador OK necesario?
Blocker Incorporar en este change No (solo notificación)
Major Incorporar salvo que el threat model lo excluya Sí, solo si rechaza
Minor Diferir a follow-up change Sí, solo si acepta ahora
Question Responder en comentario, sin code change Sí, solo si acepta como code change
Threat model gate: cada proposal.md DEBE declarar un ## Threat model nombrando el adversario y qué está fuera de scope. Findings fuera de threat model se auto-rechazan, con razón de una línea en design.md §Rejected findings.

Ejemplo aplicado — supabase-backup-policy

## Threat model (declarada en proposal.md)
Adversary: opportunistic cloud-storage breach.
Out of scope: local forensic adversary with physical disk access
            — therefore `cipher /W`, VSS clear, Tails live-USB findings auto-rejected.

4. Async / paralelo — cuándo aprovechar subagentes en paralelo

Paralelo ≠ más barato. 3 subagentes 5 min = 3× tokens de 1 subagente 5 min. Paralelo compra wall-clock cuando estás bloqueado, no presupuesto. Default: secuencial. Opt-in a paralelo en Gate D.

Fase 2 — Explorar
Research fan-out
Múltiples Explore subagentes en distintas sub-preguntas.
Single message → N Agent calls in parallel
Fase 5 — Implementar
Tasks independientes
Si tasks.md tiene grupos sin overlap (ej: endpoint A vs endpoint B), spawn N backend-developer con isolation: "worktree". Reconciliar mergeando branches.
Fase 6 — Adversarial review
Background mientras drafteas
Spawn adversarial-reviewer con run_in_background: true. Mientras corre: redactar commit message, actualizar Notion, draftear /update-docs deltas.
Fase 8 — Cierre
Sync + docs en paralelo
/openspec-sync-specs y /update-docs tocan archivos distintos. Dos Agent calls en paralelo (cuidado con conflictos si el change tocó SPEC.md y specs/).
Cuándo NO paralelizar: si quieres ver el trabajo pasar para poder interrumpir, secuencial es mejor — más fácil interjeccionar.

6. Cuándo interrumpir

Si Claude no se detiene en un gate, interrúmpelo y dilo.

Los gates son el contrato. Saltarse uno sin OK del operador es un bug del proceso, no una feature. Frases útiles:
  • "Espera — antes de design.md, lista las decisiones abiertas." (Gate A)
  • "Antes de tasks.md, muéstrame el plan en bullets." (Gate B)
  • "No spawnees backend-developer todavía — confirma el batch primero." (Gate D)
  • "Findings: dame solo las excepciones a la política default." (Gate C)
